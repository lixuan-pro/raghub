# Week 1 项目骨架讲解稿

## 1. 这个项目现在在做什么

RAGHub 是一个本地文档检索增强问答系统。

它的目标是把本地文档导入系统，并通过后续的切块、检索和大模型生成流程，实现基于本地资料的问答能力。

当前项目还处于 Week 1 工程骨架阶段，重点不是实现 RAG 核心功能，而是先搭建一个稳定、可运行、可测试、可继续扩展的后端项目基础。

## 2. 为什么第一周先做骨架、配置、日志和测试

第一周先做工程骨架，是为了避免后面直接堆功能导致项目结构混乱。

配置模块用于统一管理项目名称、版本号、运行环境和日志等级。

日志模块用于观察系统启动和运行状态。

基础测试用于保证后续修改代码时，最基本的接口不会被破坏。

这样做的核心思路是：先把项目底座搭稳，再进入文档导入、切块、检索和问答功能开发。

## 3. 当前已经完成了什么

当前已经完成 FastAPI 最小服务骨架，并提供了两个基础接口：

- `/health`：用于检查服务是否正常运行
- `/version`：用于返回当前项目版本信息

同时，项目已经完成：

- 虚拟环境与依赖管理
- README 初版
- `.env.example` 配置示例
- `config.py` 配置模块
- `logger.py` 日志模块
- FastAPI lifespan 启动日志
- pytest 基础测试
- main 和 develop 分支管理
- Week 1 周志
- 论文材料占位目录
- 评测数据占位文件

## 4. 后面会怎么进入文档导入

Week 1 收口完成后，下一阶段会进入文档导入模块。

后续会先实现 TXT 文档读取，再实现 PDF 文档读取，然后设计统一的 Document 对象，把不同来源的文本统一成后续可以处理的数据结构。

在文档导入完成之后，项目会继续进入文本切块、向量检索、RAG 问答接口和评测记录等阶段。

## 5. 当前边界

当前阶段暂时不做以下内容：

- TXT loader
- PDF loader
- Document 对象
- 文本切块
- embedding
- 向量库
- `/chat` 接口
- 前端页面
- Docker 部署

这些内容会在后续阶段逐步实现。
